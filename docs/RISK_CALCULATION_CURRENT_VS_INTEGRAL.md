# Как у нас считается риск: текущая логика и интегральная формула

## 1. Как сейчас считается риск

### Охват: не весь мир

- **Города:** риск считается только для **~71–85 городов** из внутренней базы `CITIES_DATABASE` (`apps/api/src/data/cities.py`). Произвольный город или страна без города в базе **не получают** расчёт риска.
- **Страны:** риск по стране считается как **агрегат по городам этой страны** из той же базы (`country_risk.py`). Если в базе нет ни одного города страны — возвращается заглушка (composite_risk=0.5, risk_level=medium).
- **События:** система не перебирает «все события по всему миру». Используются:
  - статические зоны (seismic zone, climate zone, political region) из базы городов;
  - при наличии — внешние API (USGS, Open-Meteo, GDELT, World Bank) только для этих городов.

Итог: **привязка региона/города к типу и уровню риска делается только для фиксированного набора городов**, а не по «анализу всего мира».

### Учёт конфликтов и событий (война, ракеты, атаки)

- **Источник:** GDELT DOC API — поиск новостей по региону с ключами `conflict`, `violence`, `attack`, `military`, `war`, `terrorism`. Агрегатор (`risk_signal_aggregator`) запрашивает GDELT по имени города/региона; калькулятор v2 использует эти сигналы с гистерезисом.
- **Охват городов:** в базе есть **Дубай и Абу-Даби (ОАЭ)**, **Доха (Катар)**, **Тегеран (Иран)**. Для них при наличии ответа GDELT конфликтный сигнал попадает в расчёт риска. Остальные города — из того же фиксированного списка.
- **Ограничения:** GDELT часто даёт таймаут; ответы не в реальном времени. UCDP/ACLED в документе «внешние БД» не интегрированы — при их подключении учёт конфликтов можно усилить.
- **Итог:** события вроде «ракеты в Катар/Эмираты», «война с Ираном» **по задумке учтены**, если приходят новости по этим регионам в GDELT и запрос не падает по таймауту. На практике обновление может запаздывать до стабилизации GDELT или добавления UCDP/ACLED.

### Текущая формула (по каждому городу)

Используется **взвешенное среднее по факторам**, без явных Severity × Probability × Impact и без гео/бизнес-весов:

```
risk_score = Σ (factor_value_i × weight_i × confidence_i) / Σ (weight_i × confidence_i)
```

- **Факторы (v1):** seismic, flood, hurricane, political, economic, infrastructure, historical.  
- **Веса (сумма 1.0):** seismic 20%, flood 18%, hurricane 15%, political 12%, economic 15%, infrastructure 10%, historical 10%.  
- **Факторы (v2):** climate, conflict, political, economic, sanctions, logistics, infrastructure — с другой матрицей весов и учётом качества (quality-aware) и гистерезиса.

Нет отдельных шкал: Severity (1–4), Probability (1–5), Impact (1–5), W_country, W_city, W_influence, Control.

### Зоны (сейчас)

| Уровень   | Условие (score 0–1) |
|----------|----------------------|
| Critical | score ≥ 0.80         |
| High     | 0.60 ≤ score < 0.80  |
| Medium   | 0.40 ≤ score < 0.60  |
| Low      | score < 0.40         |

Итоговый показатель — число в [0, 1], а не процент от максимума суммы по всем рискам.

## 2. Предлагаемая интегральная формула

Интегральный риск как **взвешенная сумма по всем рискам** с явными Severity, Probability, Impact и весами (гео + влияние + контроль):

### По каждому риску i

```
Score_i = Sev_i × Prob_i × Impact_i × W_country,i × W_city,i × W_influence,i × (1 − Control_i) × Q_i
```

(Q_i ∈ [0.6, 1.0] — качество данных.)

### Интегральный индекс (%)

```
RiskIndex% = 100 × Σ Score_i / Σ MaxScore_i
```

где **MaxScore_i** — максимально возможный вклад риска i (без контроля, Q=1):

```
MaxScore_i = 4 × 5 × 5 × W_country,i × W_city,i × W_influence,i × 1
```

(максимумы: Sev=4, Prob=5, Impact=5, Control=0.)

### Пороги зон (ваш вариант)

| Зона      | RiskIndex% |
|-----------|------------|
| Low       | 0–25%      |
| Medium    | 25–50%     |
| High      | 50–75%     |
| Critical  | 75–100%    |

### Шкалы (рекомендованные)

- **Sev:** Low=1, Medium=2, High=3, Critical=4.  
- **Prob:** 1 Rare → 5 Almost certain.  
- **Impact:** 1 Minor → 5 Severe.  
- **W_country:** от 0.90 (низкий риск страны) до 1.50 (экстремальный).  
- **W_city:** от 0.95 (спокойный) до 1.25 (критичный).  
- **W_influence:** от 0.80 (низкое влияние на бизнес) до 1.50 (критичное).  
- **Control:** 0–1 (0 = нет контролей).
- **Q_i (качество данных):** 0.6–1.0 из confidence / freshness / coverage (в формуле: × Q_i). Реализовано в `integral_risk.py`.

Таблицы весов и пороги зафиксированы в модуле `integral_risk.py` (см. ниже).

## 3. Методологические риски и обязательные доработки

### 3.1 Ключевые риски

| Риск | Описание | Рекомендация |
|------|----------|--------------|
| **Country fallback = 0.5** | При отсутствии городов в базе возвращается composite_risk=0.5, risk_level=medium. Это не «нейтрально», а искусственно задаёт риск. | **Unknown + флаг качества:** не подставлять 0.5; возвращать `risk_known: false`, `risk_level: "unknown"`, отдельный флаг качества (см. п. 3.2). |
| **Потеря confidence/quality-aware** | В v2 city risk используется учёт качества (quality-aware); в интегральной формуле изначально не было явного Q. | **Добавлен Q_i** в формулу и в `IntegralRiskInput` (диапазон 0.6–1.0). |
| **Двойной учёт географии** | W_country / W_city и факторы political / economic / conflict могут считать одно и то же дважды. | При маппинге старых факторов в Sev/Prob/Impact **не дублировать** гео в весах и в факторах: либо гео только через W_country/W_city, либо только через отдельные риски «политический/экономический» с единичными гео-весами. Зафиксировать выбор в дизайне. |
| **Нормализация через MaxScore_i** | Гео-веса в итоге работают как **веса важности** между рисками, а не как абсолютное усиление риска. | Явно зафиксировано как дизайн-решение в docstring `integral_risk.py` и в этой секции. |
| **Пропуски входов** | Нет правил для missing Prob/Impact/Control — возможны случайные перекосы при подстановке «среднего». | **Нет подстановки 0.5/среднего.** Варианты: unknown + штраф к Q (`inputs_incomplete=True` → Q × 0.8), либо исключать риск из суммы; в API возвращать флаг качества/unknown. Реализовано: `inputs_incomplete` и `MISSING_INPUT_QUALITY_PENALTY`. |
| **Агрегация страна ← города** | Сейчас — среднее (или взвешенное по exposure города). Без экспозиции (население, выручка, активы) методологически слабо. | **Экспозиционное взвешивание:** страна = взвешенная сумма по asset_value / revenue_share / population_exposed, а не простое среднее. Реализовать при переходе на интегральную модель на уровне страны. |

### 3.2 Что обязательно добавлено в формулу

- **Качество данных Q_i:**  
  `Score_i = Sev_i × Prob_i × Impact_i × W_country × W_city × W_influence × (1 − Control_i) × Q_i`  
  Q_i ∈ [0.6, 1.0] из confidence / source freshness / coverage. В коде: поле `quality` и константы `Q_MIN`, `Q_MAX`.

- **Обработка пропусков:** нет данных → unknown + штраф к Q (через `inputs_incomplete` и `MISSING_INPUT_QUALITY_PENALTY`), без подстановки «среднего риска».

- **Экспозиционное взвешивание (требуется при агрегации):** при расчёте страны — взвешивать по asset_value / revenue_share / population_exposed, а не простое среднее по городам.

### 3.3 Пороги зон и калибровка

- Пороги 0–25 / 25–50 / 50–75 / 75–100% приняты как стартовые.
- **Их нужно откалибровать на исторических инцидентах (backtesting)**, иначе зоны останутся «красивыми», но не предиктивными. Зафиксировать в дорожной карте (калибровка после накопления данных/событий).

### 3.4 Итог перед продом

- ✅ Unknown вместо fallback 0.5 для страны без городов (см. раздел 4).
- ✅ Вернуть quality/confidence в интегральной модели через Q_i и inputs_incomplete.
- ⚠ Убрать двойной учёт географии — при маппинге факторов в интегральную формулу не дублировать W_geo и political/economic.
- ⚠ Ввести exposure-weighted агрегацию при расчёте страны.
- ⚠ Калибровка порогов по историческим данным.

## 4. Где это внедрить

- **Country risk API:** при отсутствии городов в базе возвращать **unknown**, а не composite_risk=0.5. В ответе: `risk_known: false`, `risk_level: "unknown"`, при необходимости `composite_risk: null` (см. изменения в `country_risk.py`).

- **Новый модуль** `apps/api/src/services/integral_risk.py`: расчёт Score_i, MaxScore_i, RiskIndex%, определение зоны по порогам 0–25 / 25–50 / 50–75 / 75–100.  
- **Источники входов:** для каждого риска i (climate, seismic, political, cyber, …) нужны: Sev, Prob, Impact, W_country, W_city, W_influence, Control. Их можно:
  - брать из существующих факторов и зон (маппинг из текущего city_risk в Sev/Prob/Impact и гео-веса из справочников);
  - подтягивать из внешних каталогов (EM-DAT, sigma и т.д.) при интеграции.  
- **Охват «весь мир»:** интегральная формула сама по себе не решает проблему охвата. Нужно либо расширять список городов/регионов и для каждого считать набор рисков с Sev/Prob/Impact и весами, либо подключать глобальные слои (например, по странам/сетке) и маппить их в те же входы формулы.
