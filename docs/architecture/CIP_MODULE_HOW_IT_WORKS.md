# CIP Module — как он должен работать

**Critical Infrastructure Protection** — модуль защиты критической инфраструктуры: цифровой двойник объектов (энергетика, вода, транспорт, телеком, ЦОД, экстренные службы) для моделирования каскадных отказов.

---

## 1. Как он должен работать

1. **Регистрация объектов** — оператор добавляет объекты инфраструктуры (подстанции, насосные, ЦОД, узлы связи, диспетчерские и т.п.) с типом, координатами, критичностью, мощностью/населением.
2. **Зависимости** — между объектами задаются связи «кто от кого зависит» (например: подстанция зависит от ЛЭП; ЦОД — от подстанции и телекома). Без циклов.
3. **Анализ** — по выбранному объекту можно:
   - посмотреть **зависимости** (upstream/downstream);
   - запустить **cascade risk** (сколько объектов и населения затронет отказ этого узла);
   - получить **vulnerability assessment** (уязвимость, экспозиция, устойчивость, каскадный риск).
4. **Мониторинг** — агент **CIP_SENTINEL** в фоне проверяет статусы и риски и создаёт алерты (degraded/offline, высокий cascade risk, высокая уязвимость). Алерты отображаются в общем потоке алертов платформы.

---

## 2. Что там должно быть представлено

| Элемент | Описание |
|--------|----------|
| **Список объектов** | Таблица: имя, CIP ID, тип, критичность, статус (operational/degraded/offline), координаты, страна/регион. |
| **Детали объекта** | Характеристики, описание, мощность/население, владелец/оператор, история (при наличии). |
| **Зависимости** | Upstream (от кого зависит этот объект) и downstream (кто зависит от него). Сила связи, тип. |
| **Cascade risk** | Результат расчёта: глубина каскада, число затронутых объектов, население в зоне риска, cascade risk score, список затронутых узлов по глубине. |
| **Vulnerability** | Оценка уязвимости: баллы (vulnerability, exposure, resilience, cascade_risk), количество зависимостей. |
| **Статистика модуля** | Количество объектов, количество зависимостей (в шапке или блоке статистики). |

**Сейчас в UI:** список, регистрация (модалка), кнопки Dependencies / Cascade / Vulnerability по объекту, панели с результатами. Карта и граф зависимостей — по ТЗ High, пока в списке приоритетов.

---

## 3. Какие задачи выполнять

| Задача | Кто выполняет | Как |
|--------|----------------|-----|
| Регистрировать объект | Пользователь | «Register Infrastructure» → форма (имя, тип, координаты, критичность, страна, мощность, население и т.д.). |
| Добавлять зависимость | Пользователь | API: POST /api/v1/cip/dependencies (source_id, target_id, type, strength). В UI: отдельная форма/панель «Add dependency» (сейчас можно через API). |
| Просматривать зависимости | Пользователь | Выбрать объект → «Dependencies» → показать upstream/downstream. |
| Запускать cascade risk | Пользователь | Выбрать объект → «Cascade» → запрос к API → показать глубину, затронутые объекты, население, score. |
| Оценивать уязвимость | Пользователь | Выбрать объект → «Vulnerability» → запрос к API → показать баллы и рекомендации. |
| Мониторить статусы и риски | CIP_SENTINEL (агент) | В фоне: раз в N секунд проверка статусов (degraded/offline/maintenance), высокий cascade_risk_score, высокий vulnerability_score → создание алертов в общий поток. |

---

## 4. Как должен быть наполнен

- **Минимум для демо:** 6+ объектов разных типов (энергия, вода, телеком, ЦОД, экстренные службы) и 5+ зависимостей между ними.
- **Источник данных:**  
  - ручная регистрация через форму;  
  - или **автозаполнение при старте API** и при **POST /api/v1/seed** (сид `strategic_modules_seed`): 6 объектов CIP + зависимости уже создаются.
- **Рекомендуемые типы (справочник):** power_transmission, power_distribution, water_treatment, water_distribution, telecommunications, data_center, emergency_services и др. (см. CIP types в API).
- **Критичность:** tier_1 (национальная), tier_2 (регион), tier_3 (локальная). Для реалистичного сценария — хотя бы один tier_1 (например, узловая подстанция или ЦОД).

После сида или ручного ввода модуль уже «наполнен» и готов к просмотру зависимостей, cascade и vulnerability.

## 5. Какой визуал (по ТЗ и дорожной карте)

| Визуал | Приоритет по ТЗ | Текущее состояние | План |
|--------|-----------------|-------------------|------|
| **Список объектов** | Critical | ✅ Таблица с фильтрами по типу/критичности | — |
| **Карта объектов** | High (FR-CIP-002) | ✅ Map View (см. ниже) | Точки по lat/lon, цвет по cascade/vulnerability; клик → зависимости. |
| **Граф зависимостей** | High (FR-CIP-005) | ✅ Force-Directed Graph (как Cascade Analysis) | Узлы = объекты, рёбра = зависимости; раскраска по критичности/статусу. |
| **Деталь объекта** | High | Частично (панели Dependencies/Cascade/Vulnerability) | Отдельная страница или боковая панель: все поля объекта + зависимости + последние оценки. |
| **Cascade-симуляция (таймлайн, анимация)** | High (FR-CIP-007) | Частично (результаты cascade по кнопке) | Таймлайн/анимация распространения по графу. |
| **CIP Dashboard** | High (FR-CIP-009) | Частично (полоска KPI + ссылка на алерты) | Дашборд: KPI, карта, граф, последние симуляции, лента алертов. |

### Граф зависимостей (тот же стиль, что Cascade Analysis)

- **Реализация:** Force-Directed Graph (`react-force-graph-2d`), тот же подход, что на странице Analytics → Cascade Analysis.
- **Содержимое:** узлы = объекты инфраструктуры (из GET `/api/v1/cip/dependencies/graph`), рёбра = зависимости (source_id → target_id).
- **Оформление:** сетка и радиальный оверлей, кастомная отрисовка узлов (градиент по цвету, подпись в «таблетке»), направленные рёбра со стрелками, частицы на рёбрах, zoom/pan, перетаскивание узлов.
- **Цвет узлов:** по статусу (offline/degraded/maintenance → красные/оранжевые) и по критичности (tier_1 — amber, tier_2 — blue, tier_3 — green). Легенда под графом.
- **Клик по узлу:** открывает панель зависимостей выбранного объекта.

### Карта (Map)

- **Режимы:** переключатель **2D (Mapbox)** и **3D (Cesium + Google 3D Tiles)**.
- **2D:** компонент **MapView** (Deck.gl + Mapbox). Стиль: `mapbox/dark-v11`. Точки (ScatterplotLayer); размер — по «ценности»; цвет — по риску (cascade/vulnerability).
- **3D:** компонент **CesiumGlobe** с включённым слоем **Google Photorealistic 3D Tiles** (Cesium Ion). Объекты — маркеры на глобусе; при клике камера приближается к объекту на **50, 75 или 100 м** (выбор в панели «Zoom height»). Чёткий вид объекта в контексте фотореалистичной 3D-сцены.
- **Что на карте:** объекты инфраструктуры, у которых заполнены **latitude** и **longitude** (при регистрации или в данных сида).
- **Клик по точке/маркеру:** открывает панель зависимостей; в 3D дополнительно перелёт камеры на выбранную высоту (50/75/100 м).

Итого: список ✅, карта ✅ (MapView), граф ✅ (Force-Directed, как Cascade Analysis); симуляция с таймлайном и отдельный CIP dashboard — в плане Phase 1.

## 6. Какой агент-помощник должен там быть

**CIP_SENTINEL** — агент непрерывного мониторинга критической инфраструктуры.

| Что делает | Как |
|------------|-----|
| Следит за статусом объектов | Проверяет `operational_status`: degraded, offline, maintenance → создаёт алерт (CRITICAL/HIGH/WARNING). |
| Следит за cascade risk | Если у объекта `cascade_risk_score >= 70` → алерт с рекомендацией проверить зависимости и каскад. |
| Следит за уязвимостью | Если `vulnerability_score >= 70` → алерт с рекомендацией по усилению и мониторингу. |

Алерты того же формата, что и у общего SENTINEL: **попадают в общий поток платформы** — сохраняются в `sentinel_agent.active_alerts`, отдаются по GET `/api/v1/alerts`, рассылаются по WebSocket подписчикам и через event_emitter (дашборд, Command Center). У каждого алерта может быть поле `source: "CIP_SENTINEL"` для фильтрации. Включение/выключение мониторинга — через настройки алертов (SENTINEL auto-start при первом подключении к WebSocket алертов).

**Где код:** `apps/api/src/modules/cip/agents.py` (CIPSentinelAgent), вызов из `apps/api/src/api/v1/endpoints/alerts.py` в цикле мониторинга; регистрация в общий поток — в том же цикле (`sentinel_agent.active_alerts[alert.id] = alert` перед broadcast).

## 7. Краткая сводка

- **Назначение:** регистрация объектов КИ, зависимости, оценка каскада и уязвимости, мониторинг 24/7.
- **Представлено:** список объектов, зависимости, cascade risk, vulnerability, статистика; далее — карта, граф, симуляция, CIP dashboard.
- **Задачи:** регистрация и зависимости (пользователь), анализ cascade/vulnerability (пользователь по кнопкам), алерты (CIP_SENTINEL).
- **Наполнение:** сид при старте API и при POST /seed (6 объектов + зависимости) или ручной ввод.
- **Визуал:** список ✅; карта, граф, таймлайн симуляции, CIP dashboard — по плану Phase 1 (High).
- **Агент:** CIP_SENTINEL — мониторинг статусов и высоких cascade/vulnerability, алерты в общий поток.
