# SENTINEL и Overseer: что проверяют и что могут починить

**Ожидание:** «Оба проверяют всю систему и по кнопке Fix могут сами всё починить.»

**Факт:** проверки и авто-исправления ограничены; часть проблем (например, отсутствующие API-маршруты) не проверяется и не чинится автоматически.

---

## 1. Что проверяют

### System Overseer (статус «Healthy»)

- **От чего зависит «Healthy»:** только **БД (PostgreSQL/SQLite)** и **Redis**.
  - Если оба в порядке → `health.status = "healthy"`.
- **Что ещё собирается, но не меняет Healthy:**
  - Neo4j, MinIO, TimescaleDB (отдельные статусы в `databases`);
  - внешние API (NOAA, FEMA и т.д.);
  - метрики эндпоинтов (латентность, error rate) — только по **уже выполненным** запросам;
  - стратегические модули, агенты, NVIDIA.
- **Чего нет:** нет проверки «все ли нужные API-маршруты существуют и отдают 2xx».  
  Если маршрута нет (404), Overseer об этом не знает и статус остаётся Healthy.

### SENTINEL (алерты)

- Алерты создаются по правилам Overseer (БД, Redis, circuit breakers, производительность и т.д.).
- Отдельно **не** проверяет наличие конкретных URL (например, `/climate/indicators`, `/climate/forecast`).

---

## 2. Что делает «Fix» / Run / Resolve

### В виджете Overseer: кнопка «Run» (цикл)

- Запускает один цикл: сбор снимка → оценка правил → **auto_resolve_issues**.
- **auto_resolve_issues** умеет только:
  - переподключить **Redis**, **PostgreSQL**, **Neo4j**;
  - сбросить **circuit breaker** (Redis, PostgreSQL, Neo4j, MinIO, Timescale);
  - очистить кеш (expired entries) при высокой памяти;
  - залогировать медленные/проблемные эндпоинты.
- **Не умеет:** добавлять новые маршруты в код, перезапускать процесс, чинить конфиг или баги в приложении.

### В панели алертов SENTINEL: «Resolve»

- **POST /api/v1/alerts/resolve** только помечает алерт как «решённый» (флаг в БД).
- Это не исправление причины алерта, а признание «учтено / закрыто».

---

## 3. Итог: что может и что не может система

| Ожидание | Реальность |
|----------|------------|
| Проверяется «вся система», в т.ч. все API | Проверяются БД, Redis, опционально Neo4j/MinIO/Timescale; метрики эндпоинтов — только по факту запросов. Отсутствующие маршруты (404) в расчёт Healthy не входят. |
| Fix сам починит любую проблему | Fix (Run cycle) чинит только переподключения и circuit breaker; Resolve в алертах — только пометка «resolved». Добавить маршрут или перезапустить приложение автоматически нельзя. |

---

## 4. Что уже добавлено (опционально)

- **Проверка критичных маршрутов:** если в `.env` задан **`OVERSEER_CRITICAL_ROUTES_BASE_URL`** (например, `http://127.0.0.1:9002`), Overseer при каждом цикле делает GET по маршрутам:
  - `/api/v1/health`
  - `/api/v1/climate/indicators?latitude=0&longitude=0`
  - `/api/v1/climate/forecast?latitude=0&longitude=0&days=1`
  При 404/5xx создаётся алерт «Critical API route unavailable». **Авто-фикс для этого нет** (добавить маршрут в код Overseer не может).

## 5. Идеи на будущее

- **Явная подпись в UI:** например, «Run cycle: переподключение сервисов и сброс circuit breakers» и «Resolve: пометить алерт как решённый», чтобы не создавать ожидание «починит всё подряд».
