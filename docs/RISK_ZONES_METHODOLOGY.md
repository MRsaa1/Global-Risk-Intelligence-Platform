# Методика определения зон риска (страна — город — зона)

Документ описывает: **какие факторы учитываются**, **откуда берутся данные**, **какая формула** и **как город попадает в зону** (Critical / High / Medium / Low).

---

## 1. Какие факторы учитываются

Риск города считается по **7 факторам** (все нормализованы 0.0–1.0):

| Фактор | Описание | Вес в формуле |
|--------|----------|----------------|
| **Seismic (сейсмика)** | Землетрясения: близость к зонам сейсмической активности и/или данные о недавних событиях | 20% |
| **Flood (наводнения)** | Риск наводнений: климатическая зона и/или текущая погода (осадки, влажность) | 18% |
| **Hurricane (ураганы/тайфуны)** | Ураганы, тайфуны, циклоны по климатической зоне и бассейну (Атлантика, Тихий океан, Индийский) | 15% |
| **Political (политический)** | Политическая стабильность региона, конфликты, геополитика | 12% |
| **Economic (экономическая экспозиция)** | Объём экспозиции (капитал в городе) и концентрация активов | 15% |
| **Infrastructure (инфраструктура)** | Уязвимость инфраструктуры по региону и известным рискам (энергия и т.д.) | 10% |
| **Historical (историческая волатильность)** | Количество крупных исторических событий в городе | 10% |

**Итоговый риск города** — взвешенная сумма этих факторов (с учётом confidence). Дополнительно применяются **повышающие правила** для экстремальных случаев:

- **Зоны конфликтов** (political ≥ 0.85): итоговый риск не ниже ~0.70 + доп. надбавка.
- **Политическая нестабильность** (political ≥ 0.70): надбавка к итоговому риску.
- **Кризис инфраструктуры** (infrastructure ≥ 0.75 и political ≥ 0.60): надбавка.

## 2. Формула (методика расчёта)

**Базовая формула** (взвешенное среднее с учётом уверенности по каждому фактору):

```
risk_score = Σ (factor_value × weight × confidence) / Σ (weight × confidence)
```

Веса факторов (сумма = 1.0):

- seismic: 0.20  
- flood: 0.18  
- hurricane: 0.15  
- political: 0.12  
- economic: 0.15  
- infrastructure: 0.10  
- historical: 0.10  

После этого к `risk_score` применяются описанные выше **override’ы** для конфликтов и нестабильности. Итоговое значение ограничивается диапазоном 0.0–1.0.

**Отнесение города к зоне** — по порогам итогового `risk_score`:

| Зона | Условие по risk_score | Смысл |
|------|------------------------|--------|
| **Critical** | score > 0.8 | Критический риск |
| **High** | 0.6 < score ≤ 0.8 | Высокий риск |
| **Medium** | 0.4 < score ≤ 0.6 | Средний риск |
| **Low** | score ≤ 0.4 | Низкий риск |

Счётчики на дашборде (critical_count, high_count, medium_count, low_count) — это **число городов (hotspots)** в каждой из этих зон. Они считаются в `GeoDataService.get_portfolio_summary()` по списку `CityRiskScore` после пересчёта рисков.

## 3. Откуда берутся данные (источники метрик)

### 3.1 Статические данные (всегда доступны)

- **База городов** (`apps/api/src/data/cities.py`):
  - Координаты, страна, экспозиция (млрд USD), число активов.
  - Классификации: **SeismicZone** (pacific_ring, alpine_himalayan, mid_atlantic, stable), **ClimateZone** (tropical_cyclone, monsoon, coastal_flood, temperate, arid, continental), **PoliticalRegion** (oecd_stable, oecd_moderate, emerging_stable, emerging_volatile, conflict_zone).
  - **known_risks** — заданные вручную риски по городу (earthquake, flood, hurricane, political, conflict, infrastructure, energy, geopolitical и т.д.).
  - **major_events** — список крупных исторических событий.

- **Базовые риски по зонам** (в том же модуле):
  - `SEISMIC_BASE_RISK` — по типу сейсмической зоны (например Pacific Ring 0.85, Stable 0.15).
  - `CLIMATE_BASE_RISK` — по климатической зоне (например tropical_cyclone 0.80, temperate 0.35).
  - `POLITICAL_BASE_RISK` — по политическому региону (например conflict_zone 0.90, oecd_stable 0.10).

Если внешние API не вызываются или не отвечают, используются только эти данные (и известные риски по городу).

### 3.2 Внешние API (при пересчёте с «живыми» данными)

Используются только когда при пересчёте включён флаг **use_external_data** (например после истечения кэша 6h или при `?recalculate=true`):

| Фактор | Источник | Что даёт |
|--------|----------|----------|
| **Seismic** | **USGS Earthquake Catalog** | Недавние землетрясения в радиусе 500 км за последний год; по ним считается риск (частота, магнитуда M≥4, M≥6). |
| **Flood** | **OpenWeather API** (текущая погода) | Осадки (rain_1h), влажность; по ним оценивается краткосрочный flood_risk. Без API ключа — оценка по широте/долготе. |

Остальные факторы (hurricane, political, economic, infrastructure, historical) считаются по **статике**: зоны, регионы, known_risks, exposure, assets_count, major_events. World Bank в коде упомянут, но в текущей реализации расчёта зон не используется.

### 3.3 Новости, конфликты, финансы, логистика — откуда и используется ли ИИ

**Кратко:** для **расчёта зон риска по городам** (Critical/High/Medium/Low и счётчики на дашборде) **нет** мировых новостных баз, поиска через ИИ и живых данных по конфликтам, финансам, логистике или поставкам. Климатические факторы — единственные, подтягиваемые из внешних API (USGS, OpenWeather). Всё остальное — статическая база городов и (при наличии) данные портфеля.

| Тема | Используется в расчёте зон? | Источник данных |
|------|-----------------------------|------------------|
| **Климат (сейсмика, наводнения, ураганы)** | Да | USGS Earthquake Catalog, OpenWeather API; при отсутствии — зоны и known_risks из `cities.py`. |
| **Конфликты, политика** | Да (фактор political) | Только **статическая** база: `PoliticalRegion`, `known_risks["political"]`, `known_risks["conflict"]`, `known_risks["geopolitical"]` в `cities.py`. Нет мировых баз конфликтов в реальном времени и нет поиска по новостям для зон. |
| **Финансовые проблемы** | Частично (фактор economic) | Экспозиция и число активов из **базы городов** или из портфеля (Asset). Нет внешних финансовых API (рейтинги, дефолты, ликвидность) для расчёта зон. |
| **Логистика, топливо, поставки** | **Нет** | В расчёте зон риска по городам **не участвуют**. Модуль SCSS (Supply Chain) использует данные поставщиков и маршрутов из ERP (SAP, Oracle) или введённые вручную; это не глобальные метрики. Названия зон типа "Fuel Storage", "Port/Logistics Hub" — это подписи в сценариях стресс-тестов, а не источники данных для зон. |
| **Новости / «поиск с помощью ИИ»** | Нет для зон | **NewsAPI** (newsapi.org) используется только для **обогащения контекста стресс-тестов**: подтягиваются статьи по запросу (регион + "risk OR stress OR sanctions OR conflict" и т.д.) и при необходимости — геополитические алерты. Это **поиск по ключевым словам**, не ИИ-поиск по мировым базам. Результаты **не** влияют на счётчики зон и на отнесение города к Critical/High/Medium/Low. |

**Итог (модель v1):** методика определения страны/города в зону опирается на статику (зоны, регионы, known_risks, история) и на два внешних источника только по «природе»: USGS и погода.

### 3.4 Модель риска v2 (при risk_model_version=2)

При включённой **модели v2** (конфиг `risk_model_version=2`) используются дополнительные **бесплатные** источники:

| Фактор | Источник | Обновление |
|--------|----------|------------|
| **Conflict** | **GDELT DOC API** (поиск по региону: conflict, violence, military) | 15 мин |
| **Political** | **GDELT DOC API** (protest, unrest, coup) | 15 мин |
| **Logistics** | **GDELT DOC API** (port closure, supply chain disruption, fuel shortage) | 15 мин |
| **Infrastructure** | **GDELT DOC API** (power outage, water shortage) | 15 мин |
| **Economic** | **World Bank Indicators API**, **IMF SDMX API** (inflation, GDP growth, unemployment) | 24h |
| **Sanctions** | **OFAC SDN List**, **UN Consolidated Sanctions** (XML) | 24h |
| **Climate** | USGS + **Open-Meteo** (или OpenWeather) — seismic, flood, hurricane | 3–6h |

**Формула v2 (quality-aware):**  
R(t) = Σ(w_i × s_i(t) × q_i(t)) / Σ(w_i × q_i(t)),  
где q_i — качество данных фактора (freshness, полнота).  
**Гистерезис зон:** вход в HIGH при score ≥ 0.60, выход из HIGH только при score < 0.55; аналогично CRITICAL (вход ≥ 0.80, выход < 0.75), чтобы зоны не «дребезжали».  
См. `apps/api/src/services/risk_signal_aggregator.py`, `city_risk_calculator.py` (RISK_WEIGHTS_V2, apply_hysteresis).

## 4. На чём основан метод

- **Идея:** комбинировать долгосрочные структурные риски (зоны, регионы, история) с краткосрочными сигналами (землетрясения USGS, погода OpenWeather), чтобы и базовая картина, и изменение ситуации в странах/городах влияли на зоны.
- **Веса** — эвристические, подобраны так, чтобы природные катастрофы (сейсмика, наводнения, ураганы) и экспозиция имели больший вес, политика и инфраструктура — вспомогательный контекст.
- **Пороги зон** (0.4, 0.6, 0.8) — фиксированные, задают четыре уровня: Low / Medium / High / Critical.
- **Метод не является** нормативным или регуляторным стандартом; это внутренняя методика платформы для визуализации и агрегации риска по городам и для счётчиков зон на дашборде.

## 6. Модель риска v2 (опционально)

Веса v2 (RISK_WEIGHTS_V2): climate 0.22, conflict 0.22, political 0.15, economic 0.18, sanctions 0.08, logistics 0.08, infrastructure 0.07 (сумма 1.0). При `risk_model_version=2` в конфиге используются дополнительные источники: **GDELT DOC API** (конфликты, политика, логистика, инфраструктура), **World Bank** и **IMF** (экономика), **OFAC** и **UN** (санкции), **Open-Meteo** (погода). Формула — quality-aware (учёт q_i по каждому фактору). Зоны назначаются с **гистерезисом** (вход/выход по разным порогам). Код: `city_risk_calculator.py` (RISK_WEIGHTS_V2, apply_hysteresis), `risk_signal_aggregator.py`, клиенты в `services/external/` (gdelt_client, worldbank_client, imf_client, ofac_client, open_meteo_client).
